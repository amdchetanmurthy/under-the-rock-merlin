Top Gaps by Priority

15 / 20
1. cp-mi400 and sw-security-tools: Zero CI
No workflows, no static analysis, no tests, no build verification. cp-mi400 is GPU command processor microcode. sw-security-tools contains security-critical signing tools. Both can ship broken code with no automated detection.
2. pmfw-firmware: Minimal CI with 5 PSP entries
Produces 5 PSP entries (MP1, MP5, IMU instruction + data) for power management. No lint, no SimNow in PR, test infra exists but not integrated. Coverity recently added (3x builds) but no nightly.
3. Only 5 of 16 repos run unit tests in PR gate
dcgpu-esid (CppUTest), mpio (custom x86), VBL-TEE-Drv (shared CppUTest), pcie-cld-fw (GoogleTest), ucie-fw (GoogleTest). Remaining 11 repos have no host-side unit test execution in CI.
4. MPRAS-Applets: No SimNow, no nightly, no functional test
A broken applet can merge with zero boot validation. Entirely depends on MPRAS-Kernel pipeline. PR gate only builds pcie_applet for WH SP7 -- not all applets, not all targets.
5. sriov-dr: CI exists but zero functional tests
Has Coverity + lint + build in PR, and BlackDuck in nightly. But no unit tests, no SimNow, no emulation. Only static analysis validates the code.
6. No code coverage measurement in any repo (automated)
ucie-fw has gcov in nightly; pcie-cld-fw has Bullseye manual-only; art-security has Coco (manual). No repo has automated coverage in PR gate or nightly reporting.
7. No cross-repo integration testing
Each repo tests its own binary in isolation via IFWI merge. No pipeline validates all components together in a single IFWI. No regression tracking or test result trending.

Path to Uniform Testing: Concrete Steps

18 / 20

Phase 1: Baseline (All repos must have)

  • PR gate workflow that blocks merge
  • Build verification for at least one ASIC target
  • Coverity incremental static analysis (via pr-incremental-coverity-builder.yml)
  • Lint (via lint-cpp.yml or equivalent)
  • JIRA ticket validation

Gap: cp-mi400 and sw-security-tools have NONE of this. pmfw-firmware is missing lint.

Phase 2: Functional Validation (All repos should have)

  • SimNow boot test OR emulation test on PR (at least 1 product)
  • Host-side unit tests running in PR gate (CppUTest, GoogleTest, or pytest)
  • Nightly workflow with full SimNow/emulation matrix
  • AI code review (via pr-ai-codereview.yml or NBIOFW equivalent)
