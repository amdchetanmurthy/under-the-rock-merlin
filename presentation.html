Why "Merlin"?

UnderTheRock = what's beneath theRock (ROCm). Merlin is the wizard who lived under the stone in Arthurian legend. Merlin brings intelligence to the rock.

Core guarantee

Module owners don't write tests. Merlin observes what known-good firmware does in SimNow, records it as a golden baseline, and asserts every subsequent build does the same.

IFWI SPIROM Layout 8MB (0x800000)

A/B partition design with 74 firmware entries per partition

0x000000 EFS (sig: 0x55AA55AA)84B 0x002000 L1 PSP Dir (2 entries: ISHA, ISHB)48B 0x010000 RomStrap A (NBIO hw init)8K 0x012000 RomStrap B (golden, never updated)8K 0x014000 ISH A (boot priority=2)32B 0x015000 ISH B (boot priority=1)32B 0x016000 Signature Table (7 RSA-4K sigs)7x512B 0x020000 FW Partition A -- L2 Dir + 74 entries3.9MB 0x400000 FW Partition B -- backup / update target3.9MB 0x7E0000 PSP Boot Config64K

A/B Boot Selection Flow

Power On
  |
  v
MPART bootrom reads EFS @ 0x0
  |  signature = 0x55AA55AA
  v
Parse L1 PSP Dir @ 0x2000
  |  2 entries: ISHA (0x48), ISHB (0x4A)
  v
Compare ISH BootPriority
  |  ISHA.priority=2  >  ISHB.priority=1
  v
Boot from Partition A @ 0x020000
  |
  v
Parse L2 PSP Dir ("$PL2" cookie)
  |  74 firmware entries
  v
Load FW in dependency order:
  Caliptra -> FMC -> TOS -> Drivers...
  |
  v
MPASP_FW_STATUS = 0x800F0000  (PASS)

Safe update: Write new FW to Partition B, flip ISH priorities, reboot. If B fails, bootrom falls back to A.

How SimNow Actually Works

AMD's pre-silicon simulator -- our gating engine for every firmware PR

SimNow Script Anatomy (real MI450 A+A script)

# 1. Load SoC model (BSD file)
shell.open $BSD/aransas_1P1G_W21_M112.bsd

# 2. Monitor FW status registers
monitor.AddMonitor --address 0x034000d8 \
  --size 4 --space 0.0.mpasp \
  --label MPASP_FW_STATUS
monitor.AddMonitor --address 0x03010d80 \
  --size 4 --space 4.0.mpart \
  --label MID_MPART_FW_STATUS

# 3. Load firmware (THE image under test)
spirom:0.initfile {{IFWI_PATH}}
sbios_rom:0.initfile {{SBIOS_PATH}}
NVMEdrive.SetImage "busybox.img"

# 4. Non-secure fuses (CI test mode)
mpart_fuse:*.set ART_LC_STATE_MFG0 7
mpart_fuse:*.set ART_LC_STATE_MFG1 0
  # ^^ 0 = test mode, 7 = prod mode

# 5. GPU register overrides
regsmgr.AddForce MI450_A0_MID/MP_MPASPCRU/\
  MPASP_C2PMSG_97/CONTENT 0x00000204

# 6. Start simulation
shell.go

Pass/Fail Criteria

MPASP_FW_STATUS = 0x800F0000

PSP boot complete, all stages passed. This is the canonical pass marker at register address 0x034000d8.

MPASP_FW_STATUS = 0x00000000

FMC didn't start. Immediate FAIL.

Additional Pass Markers

  • "all ESIDs completed"
  • "TOS steady state reached"
  • Error log is empty

AI Test Generation -- Without Hallucination

We observe what known-good firmware does. Then assert the next version does the same.

The 4-Stage Pipeline

1. COLLECT

Boot LKG IFWI in SimNow. Record every register value, FW status transition, log marker, and boot phase timing.

2. GENERATE

AI creates acceptance tests from observations. PSP size checks from IFWI layout. Postcodes from SimNow YAML. Registers from golden baseline.

3. GROUND

Validate every assertion against real data. Can't trace to observed behavior? Mark as UNVERIFIED (advisory only, never blocks merge).

4. LEARN

After 5 consistent observations, new register values become grounded assertions. Baselines auto-grow. Failed runs never update baselines.

Anti-Hallucination Grounding Rules

The #1 risk with AI-generated tests is hallucinated assertions. These rules prevent it.

# merlin-grounding-rules.yaml

register_assertions:
  # ONLY assert registers from golden baseline
  must_exist_in: "golden_baseline.registers"
  # NEVER invent register addresses
  never_guess_addresses: true
  unknown_handling: "unverified"

postcode_assertions:
  allowed_sources:
    - ".github/simnow/*.yaml -> postcode"
    - "golden_baselines/*/postcodes.yaml"
  mi450_pass: "0x800F0000"

fw_status_monitors:  # fixed per model
  MPART_FW_STATUS:
    address: "0x034000d8"
    space: "0.0.mpart"
  MPASP_FW_STATUS:
    address: "0x034000d8"
    space: "0.0.mpasp"
  MID_MPART_FW_STATUS:
    address: "0x03010d80"
    space: "4.0.mpart"

size_assertions:
  source: "configs/psp-entry-map.yaml"
  warn_threshold_percent: 90
  fail_threshold_percent: 100

The 5 Grounding Rules

1. Register addresses must appear in the golden baseline OR SimNow monitor list

2. Expected values must have been observed in a previous successful run

3. Log markers must have appeared in a previous successful run's log

4. PSP entry types must exist in the IFWI layout (74 documented entries)

5. Timing assertions use ±20% tolerance from golden baseline, never exact values

Hard Rule

If ANY assertion can't be grounded, it's marked UNVERIFIED. Only GROUNDED assertions block merge. AI suggestions that can't be verified are logged but never gate.

MI450 Data Paths from OpenXGMI Architecture

Color-coded traffic types that firmware must manage

RED PATH -- PCIe Traffic
  Host CPU <--> PCIe Root Port <--> MI450
  Carries: Config, Interrupts, MMIO
  Uses: VC0/VC1
  FW: fw-mpio (link training)

BLUE PATH -- GPU Cross-Domain (NPA via GPUVM)
  GPU <--> OpenXGMI <--> Remote GPU/AINIC
  Carries: nHToE traffic, GPU memory access
  Uses: GPUVM address translation, UTC L1/L2
  FW: fw-mpnht, fw-mpifoe

PURPLE PATH -- RDMA via IOMMU
  XRNIC <--> OpenXGMI <--> Remote memory
  Carries: RDMA traffic
  Uses: AT=10b bypass or AT=00b untranslated
  FW: fw-mpifoe (IFoE subsystem)

BLACK PATH -- GPU In-Domain (SPA)
  GPU <--> Same-node GPU <--> System memory
  Carries: System Address transactions
  Uses: Direct SPA addressing
  FW: fw-mpio (XGMI)

BROWN PATH -- Device Memory Return
  Return path for blue path requests
